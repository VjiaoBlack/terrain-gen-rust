# Rivers as Real Barriers

## Status: Proposed
## Pillar: 1 (Geography Shapes Everything)

## Current State

Rivers are generated by the terrain pipeline (Stage 3: hydrology) using flow accumulation
with a threshold of 150.0. The `river_mask` marks cells as river, and `compute_river_width`
assigns widths between 2.0 and 5.0 tiles using Leopold power-law scaling (targeting median
main river = 3 tiles wide). River cells are carved into the heightmap and rendered as
`Terrain::Water` with the `~` glyph.

**The problem:** River tiles are `Water`, and Water is walkable at 0.15x speed (A* cost 8.0).
Villagers treat rivers as "annoying but passable" terrain. They wade through 3-tile-wide
rivers constantly. Settlements blob outward in all directions, ignoring rivers entirely.
A river running through your map should split the world in two -- instead it's a speed bump.

This directly violates Pillar 1: "A river you can't easily cross -> settlement grows along
one bank until you bridge it."

### What we have

| Aspect | Current | Target |
|--------|---------|--------|
| River width | 2-5 tiles | Same (already good) |
| Crossing | Swim at 0.15x, cost 8.0 | Impassable without bridge or ford |
| Bridges | Don't exist | Building type, enables crossing |
| Settlement shape | Radial blob | Grows along riverbank, crosses only at bridges |
| Farm bonus | None from water proximity | Fertility bonus within 3-4 tiles of river |
| A* behavior | Routes through river (expensive) | Routes to nearest bridge (cheap) |

## Design

### 1. River tiles become impassable

Change `Terrain::Water` to non-walkable for the purpose of river crossings. This is the
single highest-impact change.

```
// tilemap.rs
impl Terrain {
    pub fn is_walkable(&self) -> bool {
        match self {
            Terrain::Water | Terrain::BuildingWall | Terrain::Cliff => false,
            _ => true,
        }
    }

    pub fn move_cost(&self) -> f64 {
        // ...
        Terrain::Water => f64::INFINITY,  // impassable without bridge
        // ...
    }
}
```

**Why not keep swimming?** Because 0.15x speed is slow but not slow enough. A 3-tile river
at 0.15x speed costs the same as ~16 tiles of grass. Villagers will swim every time the
detour to a bridge is longer than 16 tiles. That's almost always. The river stops being a
barrier and becomes a tax. We want binary: crossable (at bridge) or not crossable.

**Edge case -- ocean/lake water:** The same `Terrain::Water` is used for ocean tiles
(below `water_level`). Making all Water impassable is correct -- villagers shouldn't swim
in the ocean either. Lakes become true obstacles. This is good for geography.

### 2. Shallow fords at generation time

Not every river crossing should require a bridge. The terrain pipeline should identify
natural ford locations where the river is narrow and the terrain is flat.

**Ford placement rules:**
- River width <= 2.0 at that cell (narrow point)
- Both banks have slope < 0.04 (accessible approach)
- Maximum 1 ford per 30 tiles of river length (they should be rare and valuable)

**Ford terrain type:** New `Terrain::Ford` variant.

```
Terrain::Ford => {
    ch: '~',           // still looks like water
    fg: Color(80, 140, 220),  // slightly lighter blue
    bg: Color(40, 70, 120),   // lighter bed
    speed: 0.3,        // slow but passable (2x marsh speed)
    move_cost: 3.5,    // between marsh and mountain
    walkable: true,
}
```

Fords are discoverable -- a settlement near a ford has a geographic advantage. Fords
near the settlement spawn point should be a factor in spawn placement scoring.

### 3. Bridge building type

Bridges are the primary way civilizations overcome rivers. They are placed spanning a
river and convert water tiles beneath them into walkable `Terrain::Bridge` tiles.

**Bridge definition:**

```rust
BuildingType::Bridge => BuildingDef {
    name: "Bridge",
    cost: Resources { wood: 8, stone: 4, ..DEF_RES },
    build_time: 200,   // significant investment
    size: (1, 1),       // per-tile; auto-extends across river width
    layout: TileLayout::Single(Terrain::Bridge),
}
```

**Bridge terrain type:**

```
Terrain::Bridge => {
    ch: '#',            // sturdy planks / structure
    fg: Color(140, 100, 50),   // wood brown
    bg: Color(80, 60, 30),     // darker wood
    speed: 0.9,         // nearly full speed, slight slowdown
    move_cost: 1.1,     // slightly more than grass, much less than water
    walkable: true,
}
```

**ASCII representation (Map mode):**

Bridges read clearly against water in both horizontal and vertical orientations:

```
Horizontal river, bridge crossing north-south:
  ~~#~~
  ~~#~~
  ~~#~~

Vertical river, bridge crossing east-west:
  .~~~.
  .###.
  .~~~.

Wide river with bridge:
  ~~.#.~~
  ~~.#.~~
  ~~.#.~~
  ~~.#.~~
```

The `#` reads as "built structure over water." It contrasts well with `~` water tiles
and stands out as a crossing point when scanning the map.

**Placement rules:**
- Must be placed on a water tile that is part of a river (has `river_mask = true`)
- Auto-extends: placing a bridge on any river tile builds bridge tiles across the full
  width of the river at that point (perpendicular to flow direction)
- Both endpoints must reach walkable land (can't build a bridge to nowhere)
- Builder villagers must be able to reach at least one bank to construct it

**Construction sequence:**
1. Auto-build system identifies a river crossing need (villagers frequently pathing
   to the other side, or a resource/building target exists across the river)
2. Build site placed at the narrowest nearby river point
3. Builder walks to the bank, constructs bridge tiles one at a time from their side
4. Bridge becomes usable as soon as the first tile is placed (partial crossing improves
   incrementally, but full crossing requires all tiles complete)

### 4. Auto-build bridge logic

Bridges should not be manually placed (consistent with "roads auto-build from traffic").
The auto-build system should detect when a bridge is needed.

**Trigger conditions (any of):**
- A villager's A* pathfind fails or returns a path >3x the straight-line distance because
  a river blocks the direct route
- The traffic map shows high foot traffic on both banks of a river within 10 tiles of
  each other (villagers are walking along the river to reach a distant ford/crossing)
- A build site or resource target exists across a river from the settlement center, and
  no bridge or ford exists within 20 tiles

**Site selection:**
- Find the narrowest river point within the search area (fewer tiles = cheaper)
- Prefer locations near existing traffic (where villagers already want to cross)
- Prefer locations with flat bank approaches (not cliffs or mountains at water's edge)

### 5. Settlement growth along rivers

With rivers as true barriers, settlement shape changes dramatically. The auto-build
system should treat rivers as expansion boundaries similar to how it treats mountains.

**Riverbank affinity:**
- Farms within 3 tiles of a river get a fertility bonus (alluvial soil already exists
  in the pipeline -- `dist_to_river < 4 && slope < 0.05` produces `SoilType::Alluvial`)
- The auto-build farm placement scorer should add a bonus for river-adjacent tiles
- Huts and workshops should prefer tiles NEAR the river (water access) but not ON the
  river (that's for bridges and docks later)

**Expansion pattern:**
1. Settlement starts on one bank, grows parallel to the river
2. When same-bank resources thin out or space runs low, auto-build triggers a bridge
3. Settlement expands to the far bank, initially clustered near the bridge
4. A second bridge may form if the far-bank settlement grows large enough
5. Result: settlement has a clear "old town" (first bank) and "new quarter" (far bank)
   connected by 1-2 bridges -- reads as a real riverside town

**Anti-pattern to prevent:** Settlements building a bridge immediately at tick 0 because
there's a forest across the river. The trigger should require sustained pressure -- not
just "there's something over there" but "we keep needing to go over there and can't."

### 6. Water proximity bonus for farms

The alluvial soil system already exists in the pipeline (`dist_to_river < 4`). The
gameplay effect should be visible:

**Implementation:**
- Farm growth rate multiplied by 1.0 + 0.3 * (1.0 - dist_to_river / 4.0) for tiles
  within 4 of a river (up to 30% bonus at riverbank, tapering to 0% at distance 4)
- This makes river-adjacent farms visibly more productive
- Villagers should observe this and the auto-build system should prefer river-adjacent
  farm placement

**Visual indicator:** Farm tiles near water could use a slightly different green tint
in landscape mode (lusher) to communicate the bonus.

## Implementation Plan

### Phase A: Core barrier (changes ~3 files)

1. **tilemap.rs**: Make `Water` non-walkable, add `Ford` and `Bridge` terrain variants
   with appropriate ch/fg/bg/speed/cost values
2. **terrain_pipeline.rs**: Add ford detection pass after river extraction. Scan river
   cells for narrow + flat-bank conditions, place `Ford` terrain at qualifying locations
3. **ecs/components.rs**: Add `BuildingType::Bridge` with cost/size/layout definition
4. **Tests**: Verify A* routes around rivers to fords, bridge tiles are walkable, ford
   placement generates reasonable counts

### Phase B: Bridge construction (changes ~2 files)

5. **game/build.rs**: Add bridge auto-build detection (pathfind failure or high
   cross-river traffic). Add bridge site selection (narrowest point near demand).
   Add bridge construction logic (auto-extend across river width)
6. **Tests**: Verify bridge triggers from cross-river demand, bridge spans full width,
   A* uses bridge once built

### Phase C: Settlement shaping (changes ~2 files)

7. **game/build.rs**: Update farm placement scorer to add river proximity bonus.
   Update expansion boundary logic to treat river as boundary (like mountain)
8. **ecs/systems.rs** or **simulation.rs**: Add farm growth rate bonus from river
   proximity
9. **Tests**: Verify farm placement prefers river-adjacent tiles, settlement footprint
   is elongated along river axis rather than circular

## Open Questions

- **Should fords be seasonal?** Spring snowmelt could make fords impassable temporarily,
  forcing bridge dependence. Adds drama but complexity.
- **River ferries?** A cheaper alternative to bridges for low-traffic crossings. A single
  villager operates it, others queue. Interesting but possibly over-designed for Phase 1.
- **Can bridges be destroyed?** By raids, floods, or neglect? Creates dramatic moments
  ("the bridge is out, the far-bank farms are cut off") but requires repair mechanics.
- **Dam / water control?** Listed as a "dream" feature in game_design.md. If rivers are
  real barriers, damming becomes much more powerful -- you could flood a valley or dry
  a riverbed. Park for later.
- **Do predators respect rivers?** If wolves can't cross rivers, rivers become natural
  defensive barriers. This would be a huge strategic factor. Probably yes -- wolves
  should also need fords/shallow crossings, making river-backed settlements safer.

## Validation

The feature is working when:

1. A settlement on a map with a river through the middle grows along one bank first,
   then bridges and expands to the second bank
2. Farms cluster near the riverbank (alluvial bonus is visible in growth rates)
3. Two settlements on the same map with a river between them have different shapes
   than two settlements on a river-free map
4. Removing all bridges from a save file causes villagers to reroute dramatically
   (they can't just swim anymore)
5. The player can watch a bridge get auto-built and immediately understand why it was
   placed there (high traffic at the crossing point)
